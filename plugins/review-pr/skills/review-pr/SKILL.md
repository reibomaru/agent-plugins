# PR Review - 5つの観点からの包括的レビュー

**重要: 全ての出力・コメント・サマリーは日本語で記述すること。**

## コメントポリシー（全モード共通）

**コメントは最小限に絞ること。** 以下の原則を厳守する:

- **コメントする**: 実質的なバグ、セキュリティ問題、パフォーマンス問題、ロジックエラーなど、修正が必要な問題のみ
- **コメントしない**:
  - 既存の未解決インラインコメント（返信も新規コメントも不要）
  - PR内で既に議論されている事項
  - 自明な事項、スタイルの好み、些末な改善提案
  - 著者が認識していそうな軽微な点
  - 「良いですね」等の肯定コメント
- **判断基準**: 「このコメントがなければコードに実害があるか？」— Noならコメントしない

## ステップ0: 引数の解析とモード判定

`$ARGUMENTS` をスペース区切りで解析する:
- `REPO`: 第1引数 (例: `owner/repo`)
- `PR_NUMBER`: 第2引数 (例: `123`)
- `OWNER`: REPO の `/` より前の部分
- `REPO_NAME`: REPO の `/` より後の部分

次に、Claudeがこの PR に既にレビューを提出済みかどうかを確認する:

```bash
gh api repos/$REPO/pulls/$PR_NUMBER/reviews --jq '[.[].user.login] | any(contains("claude"))'
```

- `true` が返った場合: **フォローアップモード**を実行する
- `false` が返った場合: **通常レビューモード**を実行する

---

## 通常レビューモード

### ステップ1: コンテキスト取得

1. プロジェクトルートの `CLAUDE.md` を読み込んで、プロジェクト固有のルールとパターンを把握する
2. PRの差分を取得する: `gh pr diff $PR_NUMBER --repo $REPO`
3. PRの詳細を取得する: `gh pr view $PR_NUMBER --repo $REPO`
4. PRの最新コミットSHAを取得する: `gh pr view $PR_NUMBER --repo $REPO --json headRefOid --jq '.headRefOid'`
5. 既存のインラインコメントを取得する: `gh api repos/$REPO/pulls/$PR_NUMBER/comments --jq '.[].path + ":" + (.line|tostring) + " " + .body'`

### ステップ2: 5つのサブエージェントによる並列レビュー

以下の5つのエージェントを **並列に** 起動し、それぞれの観点からレビューを実施する。

各エージェントには以下の情報をすべて渡すこと:
- PRの差分（`gh pr diff` の結果）
- CLAUDE.mdのコンテキスト
- リポジトリ名とPR番号
- 最新コミットSHA（`commit_id` パラメータ用）
- **既存のインラインコメント一覧（同じ箇所・同じ趣旨のコメントを避けるため）**

1. **code-quality-reviewer** - コード品質レビュー
2. **performance-reviewer** - パフォーマンスレビュー
3. **test-coverage-reviewer** - テストカバレッジレビュー
4. **documentation-accuracy-reviewer** - ドキュメント正確性レビュー
5. **security-code-reviewer** - セキュリティレビュー

各エージェントへの追加指示:
- **既存コメントと同じファイル・同じ行付近・同じ趣旨の指摘はしない**
- **本当に重要な問題のみコメントする（観点ごとに0〜2件が目安）**
- 問題がなければコメントせず、エージェントの結果として「指摘なし」を返す

各エージェントは `mcp__github_inline_comment__create_inline_comment` を使って、
該当コード行に直接インラインコメントを投稿する。
インラインコメントの行番号は、diff内の `+` 行（追加行）の行番号（変更後ファイルの行番号）を使用する。

### ステップ3: サマリーの作成

各エージェントのレビュー結果を集約し、PRにトップレベルコメントとして **日本語で** 投稿する。

フォーマット:
```
## コードレビュー サマリー

### 総合評価
[全体的な所見を1-2文で]

### レビュー結果

| 観点 | 結果 | 指摘事項数 |
|------|------|-----------|
| コード品質 | ✅ / ⚠️ / ❌ | N件 |
| パフォーマンス | ✅ / ⚠️ / ❌ | N件 |
| テストカバレッジ | ✅ / ⚠️ / ❌ | N件 |
| ドキュメント | ✅ / ⚠️ / ❌ | N件 |
| セキュリティ | ✅ / ⚠️ / ❌ | N件 |

### 主要な指摘事項
[重要度の高い指摘のサマリー。指摘がなければ「特になし」]
```

## フォローアップモード

### ステップF1: コンテキスト取得

1. プロジェクトルートの `CLAUDE.md` を読み込む
2. PRの最新差分を取得する: `gh pr diff $PR_NUMBER --repo $REPO`
3. PRの最新コミットSHAを取得する: `gh pr view $PR_NUMBER --repo $REPO --json headRefOid --jq '.headRefOid'`
4. 既存のインラインコメント一覧を取得する: `gh api repos/$REPO/pulls/$PR_NUMBER/comments`

### ステップF2: 新規問題の検出のみ

最新差分に、既存のインラインコメントでまだ指摘されていない **新たな重要問題** があれば、
`mcp__github_inline_comment__create_inline_comment` で新規インラインコメントを投稿する。

**重要: 既存の未解決コメントへの返信は行わない。** コメント数のスカラー倍増を防ぐため。

判断基準:
- 既存コメントと同じファイル・同じ行付近の指摘はしない
- 新たなコミットで導入された新しい問題のみ対象
- 重要度の低い問題はスキップ

### ステップF3: フォローアップサマリーの投稿

PRにトップレベルコメントとして **日本語で** 投稿する。

フォーマット:
```
## コードレビュー フォローアップ サマリー

### 新規指摘事項
[新たに発見した問題があれば記載。なければ「新規指摘なし」]

### 総評
[全体的な評価を1-2文で]
```
